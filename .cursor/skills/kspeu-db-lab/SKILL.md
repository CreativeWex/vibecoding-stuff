# KSPEU DB LAB

## Перед началом — проверка активации

**Выполнять этот workflow только если одновременно:**

1. Пользователь **явно** просит лабораторную по базам данных (лаба по БД, выполни лабораторную по PostgreSQL, KSPEU DB LAB и т.п.).
2. В контексте есть **два вложения**: файл **`.pdf`** (методичка) и файл **`.sql`** (дамп/скрипт БД).

Если **любое** условие не выполнено — **немедленно прекратить** использование скилла, не запускать скрипты, отвечать в обычном режиме Cursor.

## Установка зависимостей

```bash
python3 -m pip install -r .cursor/skills/kspeu-db-lab/scripts/requirements.txt
```

Скилл **не** подключает и **не** запускает оформление итогового `.docx` (в том числе skill `gost-word-formatter`). Оформление по методичке вуза пользователь делает в Word самостоятельно или отдельным запросом к соответствующему skill.

## Шаг A — извлечение структуры (скрипт)

Запустить из корня проекта (подставить абсолютные пути к вложениям и каталогу артефактов):

```bash
python3 .cursor/skills/kspeu-db-lab/scripts/lab_pipeline.py \
  --pdf "/absolute/path/to/methodology.pdf" \
  --sql "/absolute/path/to/database.sql" \
  --out-dir "/absolute/path/to/lab_out"
```

Результат в `--out-dir`:

- `schema.json` — таблицы, столбцы, PK/FK, индексы (по мере разбора).
- `method_statements.json` — фрагменты SQL из PDF в **линейном порядке** документа.
- `hints.json` — эвристические подсказки сопоставления имён таблиц (`difflib`).

При ошибке скрипт завершается с ненулевым кодом и сообщением в stderr — **передать пользователю** текст ошибки и рекомендации из раздела «Ошибки» ниже.

## Шаг B — сопоставление и адаптация (агент)

### Разрешённые объекты в SQL шагов

**Источник правды** — таблицы, попавшие в `schema.json` из приложенного `.sql` (ключи объекта `tables`). Это те же объекты, что создаёт `init.sql` / дамп пользователя.

Во **всех** полях `sql` в `steps.json`:

- **Разрешено** обращаться к таблицам **только** из этого перечня (включая `JOIN`, подзапросы, `UPDATE`/`INSERT`/`DELETE`, `FOR UPDATE`, `VACUUM имя_таблицы` и т.д.).
- **Запрещено** создавать или использовать для демонстраций отдельные учебные объекты: `CREATE TABLE …`, `CREATE TEMP TABLE …`, таблицы с префиксами вроде `lab_`, `t_test`, имена из методички (`product`, `invoice`, …), если их **нет** в `schema.json`.
- **Разрешено** без таблиц: чистая логика транзакций и сессии (`BEGIN`, `COMMIT`, `ROLLBACK`, `SAVEPOINT`, `SELECT 1`, `SELECT now()`, деление на ноль в `SELECT`, `pg_advisory_lock`, системные функции), если в методичке это отдельный пример.

**Как строить шаги:** разбирайте смысл примера из PDF (блокировки, уровень изоляции, deadlock, `max(id)`, `FOR UPDATE`, транзакционный DDL и т.д.) и **воспроизводите ту же идею** запросами к **своим** таблицам и столбцам (по типам и связям из `schema.json`). Подбирайте семантически близкие таблицы: например «накладные / заказы» → сущность выдачи или платежа в схеме пользователя; «каталог товаров» → справочник книг/товаров — ориентируйтесь на FK и названия полей.

**Транзакционный DDL:** не создавайте новую таблицу; допустим вариант с **уже существующей** таблицей, откатываемый `ROLLBACK` (например `ALTER TABLE … ADD COLUMN …` с последующим откатом транзакции), если это безопасно для учебной копии БД. Если подходящего столбца/операции нет — шаг с **«Аналог не найден: …»** и пояснением.

**Нет осмысленного аналога** (пример заточен под специальную таблицу методички и на схеме пользователя недостижим без новых объектов) — пункт с пометкой **«Аналог не найден: …»** в описании; в `sql` — комментарий `-- …` или краткий запрос к допустимым таблицам, максимально близкий по смыслу.

### Порядок обработки фрагментов

Для каждой записи в `method_statements.json`:

1. Определить объекты в примере методички (таблицы, столбцы).
2. Сопоставить с `schema.json`: сначала точное имя, затем ближайшее по смыслу/структуре; учесть `hints.json`.
3. Адаптировать SQL под схему пользователя (имена, типы, кавычки PostgreSQL), **соблюдая раздел «Разрешённые объекты» выше**.
4. Если аналога нет — пункт с пометкой **«Аналог не найден: …»** в описании, поле SQL — комментарий `--` или пустая строка по согласованию с пользователем.

### Поле `notes` — пошаговое воспроизведение

Текст в Word попадает в блок **«Дополнительные инструкции»**. Всё, что вы туда выносите, оформляйте как **пошаговую инструкцию воспроизведения примера**: что сделать до и после выполнения SQL из поля `sql`, в каком порядке, какие ожидаемые эффекты (ожидание, ошибка, deadlock и т.д.). Общие фразы вроде «откройте второй терминал» без нумерации и порядка команд **не допускаются**, если для шага добавлен `notes`.

**Обязательно** добавляйте `notes`, если пример методички рассчитан на **несколько параллельных сессий** или **перекрёстный порядок** команд в разных транзакциях (два и больше окна `psql`, взаимные блокировки, демонстрация `FOR UPDATE` с ожиданием, deadlock, сценарии «транзакция 1 / транзакция 2» из PDF). В инструкции **явно** укажите:

- **Терминал 1** (или **Сессия 1**) и **Терминал 2** (**Сессия 2**) — однозначные имена для каждого окна;
- **нумерованные шаги** в **хронологическом** порядке: в каком терминале какую команду выполнить следующей (при чередовании — чередуйте подпункты: Т1 шаг 3, Т2 шаг 2 и т.д.);
- где ожидается **блокировка** или **завершение** другой сессии (`COMMIT`/`ROLLBACK`).

Для **одной** сессии, если последовательность неочевидна (несколько `SAVEPOINT`, обработка ошибки, затем `COMMIT`), тоже дайте **нумерованные шаги** в одном терминале. Если шаг — один очевидный запрос без сценария — ключ `notes` **не добавляйте** (параграф в Word не появится).

Собрать JSON шагов для отчёта (см. [reference.md](reference.md) — формат `steps.json`).

## Шаг C — сборка Word

```bash
python3 .cursor/skills/kspeu-db-lab/scripts/docx_build.py \
  --steps "/absolute/path/to/lab_out/steps.json" \
  --output "/absolute/path/to/cwd/БД_Лабораторная_работа.docx"
```

`--output` — в **текущей рабочей директории** пользователя (cwd проекта).

Итоговый файл после шага C **не форматировать** автоматически (не вызывать `gost_format.py` и аналоги).

## Формат ответа пользователю (после шагов B–C)

Кратко перечислить:

1. Путь к `БД_Лабораторная_работа.docx` (черновик содержания без принудительного ГОСТ-оформления).
2. Краткое содержание: сколько шагов, сколько пропущено с «Аналог не найден»; подтвердить, что SQL опирается только на таблицы из `schema.json`.
3. При необходимости — ключевые SQL в чате (сокращённо).

## Ошибки

| Ситуация | Действие |
|----------|----------|
| Нет PDF/SQL или нет запроса на лабу | Не использовать скилл. |
| PDF не читается / пустой текст | Сообщить: проверить файл, попробовать другой экспорт PDF; OCR не входит в скилл. |
| SQL пустой или нет таблиц после разбора | Сообщить: проверить дамп, убедиться что это PostgreSQL и есть `CREATE TABLE`. |
| Ошибка парсера на фрагменте | Сообщить номер/фрагмент из лога; предложить упростить проблемный statement или приложить исправленный .sql. |
