# KSPEU DB LAB — справка

## Зависимости

| Пакет | Назначение |
|-------|------------|
| `pypdf` | Извлечение текста из PDF (аналог задачи `pdf-parse` в экосистеме Node). |
| `pglast` | Разбор SQL PostgreSQL (libpg_query). |
| `aiofiles` | Асинхронное чтение файлов. |
| `python-docx` | Сборка `.docx`. |

## Политика ошибок (fail-fast)

- **PDF:** если после извлечения текст пустой или почти пустой (только пробелы) — критическая ошибка, код выхода ≠ 0.
- **SQL:** если файл пустой — ошибка. Если после разбора **нет ни одной таблицы** (`CREATE TABLE` / `CREATE UNLOGGED TABLE` и т.д.) — ошибка (лабораторная без схемы не имеет смысла).
- **Парсер SQL:** при ошибке `pglast` на всём файле — одна попытка fallback: грубое разбиение по `;` и разбор по частям; если снова неудача — ошибка с текстом исключения.

## Формат `schema.json`

```json
{
  "tables": {
    "имя_таблицы": {
      "columns": [
        {"name": "col", "type": "int4", "nullable": true, "constraints": ["PRIMARY KEY"]}
      ]
    }
  },
  "foreign_keys": [
    {"table": "child", "columns": ["parent_id"], "ref_table": "parent", "ref_columns": ["id"]}
  ],
  "indexes": [
    {"name": "idx_x", "table": "t", "columns": ["a"]}
  ],
  "functions": [{"name": "fn", "args": "..."}],
  "triggers": [{"name": "tr", "table": "t"}]
}
```

Поля `functions` / `triggers` заполняются частично, если конструкции распознаны парсером.

**Разрешённые имена таблиц в шагах отчёта:** только ключи из `tables` (результат разбора приложенного `.sql`). Агент не добавляет в `steps.json` SQL с `CREATE TABLE`, `TEMP TABLE`, учебными таблицами методички и любыми именами, отсутствующими в `schema.json`. Смысл примеров из PDF переносится на ближайшие по роли таблицы пользователя; операции без обращения к таблицам (`BEGIN`, `SELECT 2/0`, advisory locks и т.п.) допустимы. Подробнее — в [SKILL.md](SKILL.md), шаг B.

## Формат `method_statements.json`

```json
{
  "statements": [
    {
      "index": 0,
      "char_offset": 1205,
      "section_hint": "3.2 Примеры запросов",
      "sql": "SELECT * FROM employees;",
      "source": "keyword_scan|fenced_block"
    }
  ],
  "pdf_text_length": 50000
}
```

`source`:

- `fenced_block` — блок, похожий на код (строки с отступом или после ключевых слов).
- `keyword_scan` — фрагмент, начинающийся с известного SQL-ключевого слова.

## Формат `hints.json`

Для каждого уникального идентификатора из методички (эвристика: имена после FROM/JOIN/INTO/UPDATE/TABLE):

```json
{
  "identifiers": ["employees", "departments"],
  "suggestions": {
    "employees": [
      {"schema_table": "staff", "score": 0.72}
    ]
  }
}
```

Оценка `score` — `difflib.SequenceMatcher` (0..1). Агент не обязан слепо следовать подсказкам.

## Формат `steps.json` (вход для `docx_build.py`)

```json
{
  "title": "Лабораторная работа по базам данных",
  "steps": [
    {
      "description": "Выборка всех записей из таблицы сотрудников.",
      "sql": "SELECT * FROM staff;"
    },
    {
      "description": "Блокировка строк в двух сессиях (аналог примера из методички).",
      "sql": "BEGIN;\nSELECT * FROM orders WHERE id = 1 FOR UPDATE;\n-- ...",
      "notes": "Подключитесь к одной и той же БД в двух терминалах psql.\n1) Терминал 1: выполните первую строку из SQL (BEGIN).\n2) Терминал 1: выполните SELECT … FOR UPDATE по id = 1 — транзакция держит блокировку.\n3) Терминал 2: выполните тот же SELECT … FOR UPDATE по той же строке — сессия зависнет в ожидании.\n4) Терминал 1: COMMIT; — блокировка снимается, в терминале 2 запрос завершится.\n5) Терминал 2: COMMIT;"
    }
  ]
}
```

Поле **`notes`** (необязательное) — **пошаговая инструкция воспроизведения** примера для блока «Дополнительные инструкции» в Word. Содержание должно быть конкретным (нумерация шагов, порядок действий); для сценариев с **несколькими транзакциями/терминалами** обязательно указывайте **Терминал 1 / Терминал 2** (или **Сессия 1 / 2**) и **хронологию** — в каком окне какую команду выполнить следующей. Подробная политика — [SKILL.md](SKILL.md), шаг B, подраздел «Поле notes».

Если пояснений не нужно (один очевидный запрос): **не указывайте** `notes` или оставьте пустую строку — параграф в `.docx` **не добавляется**. Значения вроде «Не требуется» трактуются как отсутствие инструкций (совместимость со старыми `steps.json`).

Многострочный текст в `notes` в JSON задавайте через `\n` в одной строке или многострочную строку в редакторе; переносы сохраняются в документе.

## Ограничения

- Все исполняемые обращения к данным в `steps.json` должны использовать **только** таблицы из `schema.json`; воспроизведение методички — через аналогичные по смыслу запросы к этой схеме.
- Текст из PDF может терять разметку; SQL в двух колонках может склеиваться — агенту может понадобиться ручная правка `method_statements.json` или дополнение шагов.
- `pglast` не поддерживает произвольный синтаксис других СУБД — только PostgreSQL.
- Семантическое сопоставление «employees → staff» выполняет агент; скрипты дают структуру и подсказки.
